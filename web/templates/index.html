登录向导

给第一次安装的用户准备的，尽量按“看一眼就会”的方式写。日常不需要反复操作这里。

🔑 进入登录页面
什么时候需要重新登录？
首次安装后需要登录一次；之后只要 Google 登录态没过期，daemon 就会持续自动刷新。通常不需要天天管。
1
启动登录浏览器

点击上方“启动登录浏览器”。服务器上的 Chrome 会在虚拟桌面里启动，并打开 Google Labs Flow 页面。

2
进入登录页面

点击“进入登录页面”，你会看到嵌入式 noVNC 页面。直接在里面像普通浏览器一样点、输、登录就行。

3
确认已经进入 labs.google

登录成功后，确保浏览器停留在 Google Labs / Flow 页面。不要关掉这个浏览器窗口。

4
回到仪表盘点击“立即刷新 Token”

如果看到最近结果变成 成功，说明登录态已经被正确同步进 Flow2API，后续就交给 daemon 自动维护。

配置

这里放真正需要填的东西，并对容易混淆的项目做防呆提示。

Connection Token 是什么？怎么获取？
它是 Flow2API 插件连接配置里的 token 字符串，不是 URL。

正确示例：abc123xyz
错误示例：http://127.0.0.1:38000/api/plugin/update-token

基础参数

示例：http://127.0.0.1:38000 或你自己的域名地址。
这里只填 token 字符串，不要填写 /api/plugin/update-token 接口地址。
登录态就保存在这里。换目录会导致需要重新登录。
Host Agent 通过这个端口附着到正在运行的 Chrome。
Linux 虚拟桌面 display，通常保持 :99 就行。
建议 15–60 分钟。太短会增加浏览器唤醒频率，太长则遇到登录态波动时恢复会慢一些。
如果从手机或另一台电脑访问，这里不要填 localhost，应填服务器实际可访问地址。

帮助 / 原理 / 稳定性

把“它到底怎么工作”“会不会越跑越重”“需不需要定时重启”这些问题都说透。

工作原理

1. 浏览器登录态留在宿主机

Google Labs 的登录态保存在服务器上的 Chrome Profile 中，不塞进 Flow2API Docker 容器里。

2. Host Agent 通过 CDP 附着到 Chrome

不是读加密数据库，也不是装浏览器扩展，而是通过 Chrome 的远程调试接口读取运行中的 cookie。

3. 读取到 session token 后推送回 Flow2API

通过 Flow2API 已有的插件更新接口，把最新登录态同步进去。Connection Token 就是这一步的授权凭证。

资源与长期运行

当前判断：Python daemon 和 Web UI 自身都很轻；真正的大头通常是 Chrome 进程，尤其是 GPU/renderer 相关子进程。
Web UI 进程内存约 60MB
Daemon 进程内存很低
主要资源热点Chrome GPU/renderer
优化方向禁用 GPU / 降低后台开销

会不会越用越大？

理论上任何长期运行的浏览器都有积累开销的可能，尤其是在云服务器 + Xvfb 环境里。所以要把风险主要看作“Chrome 的长期稳定性”，而不是 Python 逻辑。

需不需要每天定时重启？

可以做，而且是合理的防御性策略。如果你追求长期省心，可以加一个 systemd timer，每天低峰时段重启一次 browser/daemon/UI 服务，减少长期累积状态带来的偶发问题。

推荐运维策略

普通用户默认策略

刷新频率设置为 30 分钟；只有首次安装或登录失效时手动介入；其余时间交给 daemon 自动跑。

追求稳定的策略

额外加一个每天凌晨的重启计划，例如每日 04:30 重启一次 browser + daemon，可进一步减少长跑累积状态。

异常排查顺序

先看仪表盘的分层状态 → 再看登录页是否仍处于 Google Labs 已登录状态 → 再确认 Flow2API Connection Token 是否一致。
